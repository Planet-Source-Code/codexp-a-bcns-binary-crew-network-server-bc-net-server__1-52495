A. Einleitung

Der BCNS Server ist ein Netzwerk basierter Modul-Manager.
Seine Aufgaben sind Modulenverwaltung und Nachrichtenaustausch zwischen Benutzer (Client) und den geladenen Modulen. Diese Nachrichten werden an bestimmte Module weitergeleitet und da ausgeführt, wobei auch jedes Modul in der Lage ist solche Nachrichten zurück an den Benutzer zu senden.

Diese SDK beschreibt den Umgang mit der Modul Schnittstelle von BCNS Server und enthält ein Beispiel Projekt. Hier erfahren Sie wie man ein Modul für BCNS Server schreiben und implementieren kann. BCNS Module sind vergleichbar mit Plugins. Module werden in den BCNS Server geladen und nachdem sie gestartet wurden, den Clients zu Verfügung gestellt.

1. BC Net Server Modul Schnittstelle

Der BCNS Server besitzt eine Modul Schnittstelle, die es ermöglicht Daten zwischen dem Server und Modulen auszutauschen. Hierbei wird die COM/ActiveX Schnittstelle des Betriebsystems Microsoft Windows verwendet. Die Module werden als Objekte in den Server geladen und danach gestartet. Die Module lassen sich ein- und ausschalten, wodurch einzelne Module temporär deaktiviert werden können.

BCNS Modul Schnittstelle ist mit Funktionen ausgestattet, die es ermöglichen Daten auszutauschen, oder bestimmte Aktionen auszuführen. Mit Hilfe von dieser Schnittstelle lassen sich Client-Verbindungen steuern und Nachrichten zwischen Modul und Benutzer austauschen.

Diese Schnittstellen-Funktionen sind unter Punkt 2. "BC Net Server Modul Referenz" detailliert aufgelistet.


2. BC Net Server Modul Referenz

BCNS Server Module sind Microsoft Windows ActiveX DLL-Dateien, die eine globale Klasse namens "Module" mit Schnittstellen-Funktionen bereitstellen.
Jedes Modul muss intern eine globale Klasse namens "Module" besitzen. Diese Klasse sollte bestimmte Schnittstellen-Funktionen besitzen, die den Datenaustausch und den Gebrauch von Server-Ereignissen ermöglichen.
Zusätzlich muss ein Objekt vorhanden sein der einen Referenz-Objekt auf die BCNS Server Modul Schnittstelle darstellt. Dieses Referenz-Objekt wird an das Modul, beim Aufruf der StartModule() Aktion, als einziger Parameter übergeben.

Hier werden diese so genannte Schnittstellen-Funktionen genau beschrieben.

Die Schnittstellen-Funktionen sind in drei Typen aufgeteilt:

1. Eigenschaften: Funktionen die einen Wert zurückgeben müssen (Function oder Property).
2. Ereignisse: Funktionen die auf bestimmte Ereignisse reagieren (Sub oder Function).
3. Aktionen: Funktionen die eine bestimmte Aktion durchführen (Sub oder Function).

Alle folgende Schnittstellen-Funktionen sollten in der globalen Klasse "Module" vorhanden sein und werden ausschließlich vom Server aufgerufen:

Schnittstellen-Funktionen des Moduls:
Name Typ Beschreibung
StartModule Aktion Wird aufgerufen, um das Modul zu starten.
StopModule Aktion Wird aufgerufen, um das Modul anzuhalten.
Execute Ereignis Tritt auf, wenn eine Nachricht vom Benutzer erhalten wurde.
Connected Ereignis Tritt auf, wenn ein Benutzer verbunden wurde.
Disconnected Ereignis Tritt auf, wenn ein Benutzer getrennt wurde.
Started Eigenschaft (Bool) Gibt einen Wert zurück, der bestimmt, ob das Modul geladen ist.
ModuleName Eigenschaft (String) Gibt den Namen des Moduls zurück.
ModuleVersion Eigenschaft (String) Gibt die Versionsnummer vom Modul zurück.
Tabelle 2.1: Schnittstellen-Funktionen des Moduls

Alle folgende Schnittstellen-Funktionen werden vom BCNS Server zu Verfügung gestellt und können Modul-weit beim Referenz-Objekt aufgerufen werden:

Schnittstellen-Funktionen des BCNS Servers:
Name Typ Beschreibung
Disconnect Aktion Kann aufgerufen werden, um die Verbindung zu trennen.
SendLine Aktion Kann aufgerufen werden, um Nachricht an einen Benutzer zu senden.
UpTime Eigenschaft (Long) Gibt die Verbindungsdauer in Sekunden zurück.
Version Eigenschaft (String) Gibt die Versionsnummer des Servers zurück.
HostIP Eigenschaft (String) Gibt die IP-Adresse des entfernten Rechners zurück.
HostName Eigenschaft (String) Gibt den Namen des entfernten Rechners zurück.
